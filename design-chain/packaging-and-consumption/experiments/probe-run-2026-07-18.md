# Experiment log — packaging/consumption probes, 2026-07-18

*Method: `environment-research` — real runs against the actual runtime, outputs captured verbatim so a
reviewer checks this instead of re-running. Tool: **Claude Code `claude` 2.1.214**. Observation of skill
discovery uses `claude plugin details <name>` (a fresh CLI read of a plugin's component inventory) and
`claude plugin list`; trust mechanics use plain git + the filesystem. All scratch plugins were uniquely
named `zzprobe-*`, symlinked into `~/.claude/skills/`, and removed afterward (verified: 0 residue, 22 real
methodology symlinks restored, all → agent-methodology).*

## P5 — one symlink surfaces every skill; a new skill needs no new link (the under-install fix)

Setup: a scratch skills-dir plugin (`.claude-plugin/plugin.json` + `skills/zzp5-alpha`, `skills/zzp5-beta`),
symlinked once as `~/.claude/skills/zzprobe-p5` → the external dir.

```text
$ claude plugin list        → zzprobe-p5@skills-dir   Status: ✔ loaded
$ claude plugin details zzprobe-p5
  Component inventory
    Skills (2)  zzp5-alpha, zzp5-beta         # both surfaced through ONE symlink, namespaced

# add a THIRD skill dir (zzp5-gamma) to the target — no symlink change:
$ claude plugin details zzprobe-p5
    Skills (3)  zzp5-alpha, zzp5-beta, zzp5-gamma
$ ls ~/.claude/skills | grep -c zzprobe-p5   → 1   # still exactly one symlink
```

**Result: CONFIRMED.** One symlink resolves every skill in the tier; a skill added to the target appears
with no new symlink. This is the motivating under-install failure, fixed. (`plugin details` is a fresh CLI
read, so it models a fresh session; mid-session hot-reload was neither claimed nor tested.)

## P7 — a read-only `git archive` export resists tampering (no working-tree hole)

Setup: `git archive v1 | tar -x` into a dir, write a `.pinned-sha`, `chmod -R a-w`.

```text
$ ls -a mat-v1          → .  ..  .claude-plugin  .pinned-sha  skills    # NO .git present
$ echo TAMPER >> mat-v1/skills/zzpin-skill/SKILL.md
  permission denied: …/mat-v1/skills/zzpin-skill/SKILL.md               # read-only blocks the edit
$ grep MARKER mat-v1/skills/zzpin-skill/SKILL.md   → MARKER: VERSION ONE  # content intact
```

**Result: CONFIRMED.** The export carries **no `.git`**, so the `git checkout`-in-the-target attack that
round 1 found against a live working tree is structurally absent; read-only perms block a stray edit.
(Caveat: an exit-code capture in the run measured the pipe, not the write — the `permission denied` message
and intact content are the evidence.)

## P6 — a tag→tag bump changes loaded content; the old pin is untouched

`v1` = `0bfab3a9…`, `v2` = `0ef15b8d…` (skill body MARKER differs).

```text
symlink → mat-v1 :  grep MARKER via ~/.claude/…/SKILL.md  → VERSION ONE
materialize v2 read-only; re-point symlink → mat-v2 (the bump):
                    grep MARKER via ~/.claude/…/SKILL.md  → VERSION TWO
old pin mat-v1 still:                                     → VERSION ONE
```

**Result: CONFIRMED.** A bump (re-point to a new read-only materialization) changes what the loader reads;
a consumer still on the old pin is unaffected — the bump is the only moment content changes.

## P8 — a re-pointed tag is caught by the SHA pin

```text
pinned (reviewed) SHA = 0bfab3a9…              # consumer recorded v1's SHA
$ git tag -f v1 <v2-sha>                        # attacker/token moves the tag name
$ git rev-parse v1    → 0ef15b8d…               # v1 now points elsewhere
compare rev-parse(v1) to the pinned SHA → MISMATCH → boot/bump check FAILS LOUD
```

**Result: CONFIRMED.** Because the pin of record is the SHA (not the tag name), a moved tag is detected, not
silently adopted. This is why F4's read-only + SHA pin closes the round-1 hole without signing.

## P9 — the tier-independent boot check fires on every failure mode, including absence

A standalone `bootcheck.sh` (lives outside any plugin) asserting the tier resolves, its `.pinned-sha`
matches, and each expected skill is present:

```text
(a) healthy            → OK …, 1 skill(s) present            exit 0
(b) wrong expected SHA → FAIL … DRIFTED (pinned … != …)      exit 1
(c) expected skill gone→ FAIL … PARTIAL — skill 'x' missing  exit 1
(d) symlink removed    → FAIL … MISSING (absent/dangling)    exit 1   ← still runs with the tier gone
```

**Result: CONFIRMED.** Detects drift, partial, and absent tiers loudly. Case (d) is the round-1 point: a
standalone check reports the tier's absence, whereas a `SessionStart` hook bundled *inside* the tier could
not fire when the tier is missing.
